# Security Notes

## What Is Gitignored and Why

### Secrets and Credentials

| Path | Reason |
|------|--------|
| `.env` / `.env.*` | Gmail credentials, LinkedIn session dir, post window config — never commit |
| `.secrets/` | OAuth token files, WhatsApp LocalAuth session, LinkedIn Playwright session |
| `mcp-email-server/.env` | MCP server environment variables |

These files must be created locally from `.env.example`. They are never committed.

### Runtime State

| Path | Reason |
|------|--------|
| `.state/` | Deduplication IDs (`processed_ids.json`), LinkedIn scheduler state — machine-specific, not portable |

### Vault Data (AI_Employee_Vault/)

All vault subdirectories except `Dashboard.md` and `Company_Handbook.md` are gitignored:

| Path | Reason |
|------|--------|
| `Needs_Action/*.md` | Contains real message content from Gmail and WhatsApp |
| `Pending_Approval/*.md` | Draft replies awaiting human review — may contain private content |
| `Approved/`, `Rejected/` | Execution-ready or cancelled actions |
| `Plans/*.md` | AI-generated plans and drafts |
| `Done/*.md` | Completed items — may contain personal data |
| `Logs/*.json` | Daily audit logs with message metadata |

`Dashboard.md` and `Company_Handbook.md` are committed because they are configuration/status files with no runtime data.

### Build Artifacts

| Path | Reason |
|------|--------|
| `.venv/` | Recreated by `uv sync` |
| `mcp-email-server/node_modules/` | Recreated by `npm install` |
| `__pycache__/`, `*.pyc` | Python bytecode — not portable |

## PHR Files (../history/prompts/002-silver-tier/)

PHR (Prompt History Records) are development session logs generated by the SDD workflow.

The entire `history/prompts/002-silver-tier/` directory is **gitignored** (root `.gitignore`).

**Why**: Live testing sessions produced PHR files containing real contact names, real email addresses, WhatsApp contact IDs, and Gmail message IDs from actual end-to-end pipeline tests. These are internal audit logs, not code artifacts. Committing them to a public repo would expose third-party contact data.

What was inside (not published):
- WhatsApp session logs with real contact names and phone number hashes
- Gmail triage/reply logs with real sender emails and message subjects
- LinkedIn draft session logs referencing real business activity

PHRs for other tiers follow the same rule: if live session data was captured during testing, gitignore that feature's PHR directory.

## Secrets Setup Checklist

Before running the system, ensure these are in place:

- [ ] `.env` created from `.env.example` (fill all values)
- [ ] `.secrets/gmail_credentials.json` obtained from Google Cloud Console
- [ ] `.secrets/whatsapp_session/` created by first-run QR scan
- [ ] `.secrets/linkedin_session/` created by first-run login
- [ ] `mcp-email-server/.env` configured with same Gmail credentials

## No Hardcoded Credentials

All code reads credentials exclusively from environment variables and `.secrets/` files. No tokens, passwords, or API keys appear in committed code. Test files use placeholder values (`x@y.com`, `alice@example.com`).
